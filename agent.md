# 织 · Loom —— AI 协作与开发规范 (agent.md)

本规范文件旨在明确 AI 助理（Antigravity）在开发“织 · Loom”项目时的**职责范围**、**AI 与人类的协作决策边界**以及项目的**工程代码规范**。后续所有的开发代理均须 100% 遵守本规范。

---

## 一、 AI 与人类的协作决策边界

为了保障代码的可控性与安全性，AI 在执行任务时必须遵循以下“三级决策矩阵”：

### 1. 独立决策区 (AI 可以直接执行并提交)
对于以下低风险任务，AI 可独立编写、修改并直接运行测试：
*   **前端 UI 细节**：符合 `teenageEngineeringUI.md` 规范的 Vue 组件编写、样式调整及布局拼装。
*   **局部算法实现**：VFS 解析器内部数据处理函数、无副作用的辅助类与工具函数。
*   **测试用例扩充**：为现有模块编写额外的单元测试（Unit Tests）与集成测试。
*   **代码清理与优化**：无功能变动的 Lint 修复、多余导入清理、代码注释添加。

### 2. 方案审核区 (AI 必须先提交 `implementation_plan.md` 获得人类批准)
对于以下影响全局或涉及架构的任务，AI **禁止直接动手**，必须先在 `implementation_plan.md` 中阐述方案并等待人类输入 `Proceed`：
*   **数据库变更**：SQLite (`loom.db` 或插件独立库) 的 Schema 修改、新建表、字段增删、迁移脚本（Migrations）编写。
*   **Rust 依赖与核心变更**：修改 `Cargo.toml` 引入新 Crates，或者对 VFS 核心 Traits (`Source` 定义等) 进行非向下兼容的修改。
*   **安全性与系统权限**：修改 Tauri 的 capabilities 配置文件，放开 VFS 对磁盘或网络代理的访问控制权限。
*   **网络与流式引擎**：对基于 Range 的流式媒体服务、自定义协议、转码兜底等逻辑的重构。

### 3. 绝对红线区 (AI 绝对禁止执行)
*   **设计文档删除**：**严禁删除或覆盖**项目中的设计书（如计划书、UI 规范文档等）。任何初始化或脚手架覆盖操作必须单独建立临时隔离区。
*   **宪法违背**：禁止编写任何违背《独立开发计划书》中“一、核心愿景与设计宪法”（例如引入账号登录、硬塞云存储、使用 Electron 重新打包等）的代码。

## 二、 项目开发技术规范

### 1. 前端规范 (Vue 3 + TS + Tailwind v4)
*   **TypeScript 类型安全**：禁止使用 `any`，所有 API 返回值和 IPC 通信格式必须显式定义 Interface，且优先与 Rust 侧的数据结构（通过 `ts-rs` 等导出）保持 1:1 强类型同步。
*   **Tailwind v4 主题规范**：
    *   严禁使用 ad-hoc 的硬编码颜色值（如 `text-[#ff5c00]`）。必须统一使用 [src/index.css](file:///d:/code/rust/loom/src/index.css) 中 `@theme` 声明的自定义变量（如 `bg-bg-base`, `text-text-primary`, `text-accent`）。
    *   保持绝对纯净的扁平工业设计，禁止随意引入渐变、拟物化阴影或毛玻璃等修饰效果。
*   **等宽数字对齐**：任何展示数字（大小、比特率、时间）的表格与列表单元格，必须加上等宽数字 CSS 类，确保在 TE 风格下的精密排版效果。

### 2. 后端规范 (Rust / Tauri v2)
*   **无 panic 错误处理**：禁止在生产代码中使用 `unwrap()` 或 `expect()`。所有可能失败的操作（IO、JSON反序列化、SQLite查询）必须使用 `Result`，通过 `?` 向上抛出，并在前端层进行受控的降级提示。
*   **数据主权与路径无关性**：在 Rust 核心或 VFS 存储本地路径时，**严禁写死任何开发机的绝对路径**（例如 `/Users/username/...` 或 `C:\Users\username\...`）。所有的本地路径索引必须通过相对挂载点的逻辑路径（相对 VFS 根路径）进行映射，保障数据库在不同设备间可完美备份与无缝迁移。
*   **并发安全**：由于 Tauri 具有多线程并发特性，Rust 侧共享的状态或数据库连接句柄，必须使用安全的同步锁（如 `Arc<Mutex<T>>` 或 `tokio::sync::RwLock`），避免死锁。

## 三、 Git 提交与文档沉淀规范

### 1. 提交信息规范 (Conventional Commits)
AI 提交代码时必须严格遵守 Conventional Commits 格式。主要类型包括：
*   `feat`: 新增 VFS 驱动、插件 API 等具体功能。
*   `fix`: 修复 Rust 崩溃、前端样式对齐或内存泄漏等 Bug。
*   `docs`: 更新 `README.md`、`agent.md` 或其他规范说明。
*   `style`: 前端微调（空格、排版等），不影响代码逻辑。
*   `refactor`: 既不修复错误也不添加功能的代码重构。

每次完成一个阶段性的原子任务，必须及时执行 `git commit`，禁止一次性提交过多无关的跨模块改动。

### 2. 变更沉淀 (Walkthrough.md)
在每一次重大的功能更新或架构调整后，AI 必须创建或更新 `walkthrough.md` 记录本次的改动内容、验证方案和运行结果，作为向人类汇报以及为其他 AI 代理同步进度的依据。
