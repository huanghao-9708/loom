# 织 · Loom —— 第二阶段 (分层插件系统与沙箱生态) 实现计划文档

> 本文档基于《织 · Loom —— BYOS 文件管理平台 · 独立开发计划书》提取并细化，作为第二阶段开发过程中的指导性任务拆解与执行标准。

## 🎯 阶段目标：插件生态底座与多重数据隔离架构
**预计耗时**：约 1.5 个月
**核心边界**：本阶段**不进行业务层面强应用插件的深挖**。开发资源百分之百倾斜于打磨“系统底座通道”，构建一套让第三方 JS 脚本在绝对安全、物理隔离且无系统权限前提下运行的沙箱底盘。

**可演示最终目标**：
在宿主运行时，动态无缝加载一个演示级插件 (`com.loom.hello`)。该插件运行于受限的沙盒内，拥有自己专属并可独立建表的 `data.db`。它可以通过官方提供的 `loom-sdk.js` 与其专有的数据库与指定目录进行安全的通信交互，但任何试图读取未授权文件系统或调用原生环境 API 的跨界尝试均将被 Rust 底层防火墙严厉拦截与阻断。

---

## 🛠 任务拆解与执行清单

### 1. 宿主注册表与多实例沙箱管理 (SQLite Multi-Pool)
- [x] **宿主 `loom.db` 元数据强化**：建立 `plugins` 表，由系统宿主只记录插件的元数据（包名、版本、权限清单）而不干涉业务数据。
- [x] **系统级沙箱目录归档**：建立插件统一沙箱与运行时路径（`AppData/Local/com.loom.app/plugins/<plugin_id>/`），其内包含各自独立的前端离线资源及专用的 `data.db`。
- [x] **后台连接池池化管理 (`PluginManager`)**：在 Rust 侧建立并维护 `HashMap<String, SqlitePool>`。由统一调度器长链接并分配激活插件的数据库句柄，彻底杜绝插件污染主库的可能性。
- [x] **独立 Schema 版本控制执行器**：实现基于插件 `manifest.json` 的初始化与自动升级器，支持在对应的独立库内有序执行迁移（Migrations）SQL 文件。

### 2. 纵深防御通信矩阵 (Tauri API 与 IFrame RPC)
- [x] **Rust 核心防火墙网关 (Commands)**：针对插件特殊设计如 `cmd_plugin_db_query(id, sql)` 与 `cmd_plugin_vfs_list(id, path)` 等 Tauri 原生指令拦截层。
- [x] **前端安全载体 (`PluginHost.vue`)**：使用包含严格安全策略属性的 `<iframe sandbox="allow-scripts">` 提供前端渲染容器，完全屏蔽宿主 Vue 的上下文污染。
- [x] **防篡改的跨域消息总线**：利用前端 `postMessage` 构建 RPC 桥接服务。IFrame 内任何系统层调用必须序列化给宿主中转，宿主再根据运行时动态派发的 Token 下发给 Tauri 后端。

### 3. Loom 官方插件底层开发套件 (`loom-sdk.js`)
- [x] **极简运行时 Wrapper**：面向插件开发人员输出零外部依赖的纯粹原生 JS 框架垫片。
- [x] **DB 命名空间映射**：提供 `loom.db.query(sql, params)` 异步驱动封装。
- [x] **VFS 命名空间映射**：提供如 `loom.vfs.list(path)` 目录枚举和 `loom.vfs.streamUrl(vpath)` 的封装。

### 4. 权限白名单与拦截引擎 (Manifest Policies)
- [ ] **清装解析器**：动态校验外置的 `manifest.json` 结构的完整性，抽取其声明的白名单（例如 `"permissions": {"vfs": ["read"], "db": true}`）。
- [ ] **调用栈鉴权机制**：Rust 核心层在所有相关 RPC 访问处介入，如果插件 ID 未命中权限表，将立即引发恐慌（Panic）或返回安全拒绝级别警告 `Err(Unauthorized)`。

### 5. 概念联调与全链路验证 (PoC 闭环测试)
- [x] **Hello World 靶机编写**：制造并导入 `com.loom.hello` 测试插件及 UI 界面。
- [x] **正向流打通**：通过 SDK 完成自主 `data.db` 建表、插入一条配置并拉取。
- [x] **反向流压测**：测试插件直接调用 `window.fetch`，试图逃逸沙箱篡改主 `loom.db` 行为等危险渗透测试，以验证 Rust 守门员拦截能力。

## 🚧 阶段技术挑战与对策

1. **多线程并发池耗尽系统资源与内存泄露**：
   *应对*：采用基于 Tokio 并发锁及 LRU Cache 淘汰策略。当一个插件面板被隐藏或闲置超过特定 TTL，Rust 核心引擎将自动回放（Drop）并释放其 `SqlitePool` 与相关文件句柄。
2. **IFrame `postMessage` RPC 通信吞吐量瓶颈**：
   *应对*：如果是巨量数据（例如读取数万首音频元数据的检索返回），JSON 反序列化将极其影响沙箱内主线程响应。必要时对超长 JSON 实行流切片（Chunk）透传，并在 SDK 层内置解切片缓冲机制。
3. **恶意插件沙箱提权**：
   *应对*：严格采用运行时注入生成随机的高熵鉴权密钥 `runtime_token`。如果插件端伪造 `plugin_id` 发起 Tauri 越权请求，Rust 底层必须将传递的 Token 与挂载映射进行身份匹配。
